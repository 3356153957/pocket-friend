# Pocket Friend 双 T5AI 硬件 Demo 设计

- 日期：2026-07-24
- 状态：已确认，等待实现计划
- 目标：用两台 T5AI 设备跑通发现、震动、双向确认、同步拍照和结果显示的硬件闭环
- 适用范围：现场功能 Demo，不作为量产硬件设计

## 1. 背景与目标

Pocket Friend 的长期产品方向是手机 App 为主、智能挂件为可选外设。本阶段不实现完整 App、远程服务器或人物数据长期存储，而是优先验证两台挂件之间的核心交互仪式：

```text
联网待机 -> 发现对方 -> 震动提醒 -> 双方确认 -> 同步倒计时
-> 同时拍照 -> 显示拍摄结果 -> 返回待机
```

Demo 使用两台 T5AI-Board。每台设备均带 3.5 英寸 320x480 竖屏触摸 LCD、摄像头、一个物理按钮、TB6612FNG 电机驱动板和带偏心配重的有刷直流电机。

## 2. 范围

### 2.1 本阶段包含

- 两台设备连接同一个手机热点。
- 使用局域网 UDP 自动发现和协调捕捉流程。
- 竖屏触摸 UI。
- 物理按钮作为确认和应急操作兜底。
- 电机模拟震动反馈。
- 摄像头预览、同步拍照和本机结果预览。
- 超时、取消、掉线和摄像头失败的安全恢复。

### 2.2 本阶段不包含

- 远程服务器、MQTT 或 WebSocket。
- 手机 App 联动和小岛居民生成。
- 照片上传、TF 卡持久化或云端存储。
- AI 像素头像、人脸识别或陌生人识别。
- UWB、BLE 测距或真实距离计算。
- 电机 PWM 强度控制。

照片只保留在 PSRAM 中用于当前结果页，重启或下一轮拍摄后允许丢弃。

## 3. 总体架构

两台设备运行同一套固件，通过配置区分 `DEVICE_A` 和 `DEVICE_B`。两台设备具有相同的屏幕、触摸、按钮、摄像头和电机能力。A 机仅在一次捕捉会话中承担协调者职责。

```text
设备 A                                      设备 B
屏幕 / 触摸 / 摄像头 / 按钮 / 电机          屏幕 / 触摸 / 摄像头 / 按钮 / 电机
                    \                      /
                     手机热点 + 局域网 UDP
```

- UDP 只传控制消息，不传实时画面或照片。
- A 机在双方确认后协调准备、倒计时和最终结果。
- B 机收到捕捉指令后按相同相对延时拍照。
- 通信层与业务状态机隔离，后续可将 UDP 替换为远程 WebSocket 或 MQTT，而不重写交互状态机。

## 4. 状态机

```text
BOOT
  -> CONNECTING
  -> ONLINE_IDLE
  -> PEER_FOUND
  -> WAITING_CONFIRM
  -> CAPTURE_PREPARE
  -> COUNTDOWN
  -> CAPTURING
  -> WAITING_RESULT
  -> SUCCESS
  -> ONLINE_IDLE
```

旁路状态包括 `CAMERA_PREVIEW`、`DND`、`RECONNECTING` 和 `ERROR`。

| 状态 | 屏幕 | 电机 | 输入 |
| --- | --- | --- | --- |
| `BOOT` / `CONNECTING` | 启动和热点连接状态 | 关闭 | 暂不响应 |
| `ONLINE_IDLE` | 像素待机首页 | 关闭 | 触摸相机/勿扰；长按按钮进入勿扰 |
| `CAMERA_PREVIEW` | 摄像头独占全屏 | 关闭 | 点击或按按钮返回 |
| `PEER_FOUND` | 显示对方设备编号和确认/拒绝 | 短震两次 | 触摸或按钮确认 |
| `WAITING_CONFIRM` | 显示双方确认状态 | 每 2 秒轻震一次 | 触摸或按钮取消 |
| `COUNTDOWN` | 全屏显示 3、2、1 | 最后一秒短震 | 忽略输入 |
| `CAPTURING` | 拍照提示 | 关闭 | 忽略输入 |
| `SUCCESS` | 显示本机照片和成功状态 | 连续震动 600ms | 完成或自动返回 |
| `DND` | 勿扰页面 | 关闭 | 长按按钮或触摸恢复 |
| `ERROR` | 错误和重试入口 | 三次短震 | 触摸重试 |

任一捕捉阶段超时、任一设备取消或掉线时，双方都终止本轮并恢复到安全待机状态。

## 5. 触摸界面

屏幕固定使用竖屏 `320x480`。第一版只实现六个必要页面。

### 5.1 待机页

- 顶部显示 Wi-Fi、本机编号和对方在线状态。
- 中部显示“在线探索”状态。
- 底部提供相机和勿扰两个大触摸按钮。
- 每个触摸目标不小于 `64x64px`。

### 5.2 摄像头预览页

- 摄像头独占屏幕，不叠加 LVGL 控件。
- 点击屏幕或按物理按钮返回待机页。
- 发现另一台设备时自动退出预览并进入匹配页。

### 5.3 匹配页

- 显示另一台设备编号。
- 提供拒绝和确认两个按钮。
- 物理按钮短按等同于确认。

### 5.4 等待页

- 分别显示“你：已确认”和“对方：等待确认/已确认”。
- 底部只保留取消操作。

### 5.5 倒计时页

- 全屏显示 `3 -> 2 -> 1`。
- 页面不接受触摸或按钮操作。

### 5.6 结果页

- 显示本机刚拍摄的照片和捕捉结果。
- 点击完成或等待 5 秒后返回待机。

不实现复杂动画、实时画面叠加、头像生成或多层导航。

## 6. 显示所有权

屏幕必须只有一个写入者。`display_manager` 管理三种互斥模式：

- `UI_MODE`：只允许 UI 绘制待机、匹配、等待、倒计时和错误页面。
- `CAMERA_MODE`：暂停 UI 刷新，只允许摄像头帧写屏。
- `PHOTO_MODE`：停止摄像头预览，显示静态拍摄结果。

模式切换必须先停止当前写入者，确认其不再刷新，再启用下一写入者。禁止摄像头与 LVGL 同时写屏，禁止保留 Hello World 演示线程。

## 7. UDP 协议

两台设备使用固定 UDP 端口，例如 `37800`。心跳和发现阶段使用广播，建立会话后优先使用单播。

| 消息 | 作用 |
| --- | --- |
| `HELLO` | 每秒广播设备编号、IP 和当前状态 |
| `CONFIRM` | 用户同意本次捕捉 |
| `CANCEL` | 用户取消、超时、掉线或进入勿扰 |
| `CAPTURE_PREPARE` | A 机要求 B 机准备摄像头 |
| `PREPARE_ACK` | B 机确认摄像头已准备好 |
| `CAPTURE` | A 机下发 3 秒相对倒计时 |
| `CAPTURED` | 报告本机拍照是否成功 |
| `SUCCESS` | 双方均拍照成功 |
| `RESET` | 会话结束并返回待机 |

每条消息至少包含：

```text
protocol_version
device_id
session_id
message_id
message_type
sequence
timestamp
```

可靠性规则：

- `HELLO` 每秒发送一次，连续 3 秒收不到则判定对方离线。
- 关键消息重复发送 3 次，间隔 100ms。
- 使用 `message_id` 去重，同一业务指令只执行一次。
- A 机收到 `PREPARE_ACK` 后才能发送 `CAPTURE`。
- 两台设备在收到 `CAPTURE` 后使用本地 3 秒定时器执行拍照。
- 任一方 10 秒未确认，双方取消本轮。

## 8. 按钮和触摸映射

| 操作 | 生效状态 | 行为 |
| --- | --- | --- |
| 触摸确认 | `PEER_FOUND` | 确认捕捉 |
| 物理按钮短按 | `PEER_FOUND` | 等同触摸确认 |
| 触摸取消或按钮再次短按 | `WAITING_CONFIRM` | 取消本轮 |
| 触摸完成或按钮短按 | `SUCCESS` | 返回待机 |
| 物理按钮长按 1.5 秒 | 待机或匹配阶段 | 在线/勿扰切换 |
| 任意输入 | 倒计时或拍照阶段 | 忽略 |

按键消抖时间为 `50ms`。触摸和按钮事件必须先转换为统一的应用动作，再交给状态机处理，不能各自直接控制摄像头、电机或网络。

## 9. 电机反馈

每台设备使用 TB6612FNG 驱动带偏心配重的有刷直流电机。第一版维持现有 P6/P7 启停控制，不实现 PWM 强度调节。

| 事件 | 震动模式 |
| --- | --- |
| 发现对方 | 120ms 开、100ms 停、120ms 开 |
| 本机确认 | 80ms |
| 等待对方 | 每 2 秒 60ms |
| 双方确认 | 200ms |
| 捕捉成功 | 600ms |
| 掉线或失败 | 连续 3 次 80ms |

安全要求：

- GPIO 初始化前和启动后首先将 P6/P7 置低。
- 拍照前至少 300ms 停止电机，避免画面模糊。
- 取消、掉线、错误和状态机复位时都必须执行电机停止。
- 内部 5V 供电仅用于小电机的短时 Demo；出现复位、USB 断连、屏幕闪烁或明显发热时立即断电。

## 10. 软件模块边界

- `app_state_machine`：唯一业务状态来源，处理事件和状态转换。
- `udp_transport`：发现、收发、重试、去重和掉线检测。
- `display_manager`：UI、摄像头和静态照片三种显示模式切换。
- `camera_service`：预览、准备、拍照和最新 JPEG 缓冲。
- `motor_feedback`：执行预定义震动节奏并提供强制停止。
- `input_manager`：将触摸和物理按钮转换为统一动作。
- `device_config`：设备编号、热点配置、UDP 端口和调试参数。

任何驱动回调都不直接改变业务状态；回调只产生事件，由 `app_state_machine` 串行处理。

## 11. 异常和降级

- 热点连接失败：显示未联网状态和重试按钮，电机保持停止。
- 对方掉线：发送或本地生成取消事件，三次短震后返回待机。
- UDP 重复或乱序：按 `message_id` 和 `sequence` 去重、丢弃旧消息。
- 摄像头准备或拍照失败：向对方发送失败结果，双方不进入成功页。
- 触摸不可用：物理按钮仍能确认、取消和返回。
- 一台设备重启：另一台在 3 秒心跳超时后退出当前会话。
- 软件重启：电机默认停止，旧 `session_id` 不恢复。

## 12. 验收标准

1. 两台设备能连接同一热点并在 3 秒内互相发现。
2. 两边同时进入匹配页面并执行两次短震。
3. 触摸确认和物理按钮确认均可完成双向确认。
4. A 机只在收到双方确认和 `PREPARE_ACK` 后发送捕捉指令。
5. 两台设备的拍照触发时间差不超过 100ms，以设备日志时间戳验收。
6. 两块屏幕都能显示各自拍摄结果，并在 5 秒后恢复待机。
7. 连续执行 20 轮，不出现 Hello World、花屏、死机或电机持续运行。
8. 任一设备取消、掉线或重启，另一台在 3 秒内恢复安全状态。
9. 拍照前电机已停止至少 300ms。
10. 所有自动化测试和 TuyaOpen 完整构建通过后才允许烧录实机。

## 13. 后续演进

硬件闭环验证完成后，可以在不修改业务状态机的前提下：

- 将 UDP 替换为远程 WebSocket 或 MQTT。
- 将照片由 PSRAM 改为 TF 卡或通过 HTTP 上传。
- 接入手机 App、小岛居民和像素头像生成。
- 将普通电机替换为更适合可穿戴设备的扁平震动马达。
- 增加 BLE 接近判断、低功耗、电量管理和不可关闭的拍照状态灯。
